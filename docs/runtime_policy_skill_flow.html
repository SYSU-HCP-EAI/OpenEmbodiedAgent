PolicySkillRuntime 执行通路

从上层提交 session 指令，到 watchdog 选择任务、运行 preflight、构建 target 与 policy client，再进入观察-推理-动作闭环。

配置与指令 watchdog 调度 session/runtime 边界 policy 推理链路 target 执行链路 结果落盘

端到端总览

上层输入

SESSIONS.md

用户任务进入 pending session：包含 task_description、target_ref、skill_ref、超时、执行步数和 policy_endpoint。

SKILLS.md

通过 runtime_kind: policy 与 runtime: OpenPISkillRuntime 选择 policy 路径，并声明 policy adapter 与动作契约。

TARGETS.md

指定 target runtime、target adapter、supported skills、sensor/observation/action 约束。

→
调度与检查

WatchdogSupervisor.run_once()

读取 runtime 文档，调用 scheduler 按优先级选择 pending session，并通过 registry 抢占 session。

SessionScheduler.resolve_session()

解析 target:// 和 skill://，检查 target 是否支持 skill，以及 skill 是否支持 target kind。

RuntimeCompatibilityPreflight.check()

校验 policy endpoint、policy adapter、target adapter、action bridge、sensor、动作契约，并产出 AdapterPlan。

→
运行时装配

构建 target

TargetRuntimeRegistry.build() 根据 target class 选择本地 target 或 remote proxy。

构建 policy client

build_policy_client() 根据 dummy://、openpi:// 或 policyws:// 创建 websocket policy client。

构建 skill runtime

SkillRuntimeRegistry.build() 按具体 SkillRuntime （如 OpenPISkillRuntime） 创建具体 policy skill runtime。

→
SessionRunner

SessionRunner.start()

构建 adapter stack，依次执行 target.build()、configure_session()、start_session()、reset()。

TargetSessionHandle

把 target 原始访问收敛成受控接口：observe()、action_chunk()、execution_status()、environment refresh。

进入 policy 分支

当 runtime_kind == "policy" 时，检查 runtime 类型并调用 run_policy_loop()。

→
闭环与结果

OpenPISkillRuntime.run_policy_loop()

在 max_steps 内执行观察、policy 输入转换、远端或本地推理、policy 输出转换。

动作下发

TargetSessionHandle.action_chunk() 执行 action bridge 和 target adapter 转换后调用 target.action_chunk()。

结果落盘

成功、done、失败或超时后生成 SkillRuntimeResult，再由 watchdog 写 episode、session history，并更新 session 状态。

Policy 闭环内部步骤

1

target_handle.observe()

复用 reset 返回的初始 observation，之后调用 target.observe()，再通过 target adapter 生成 runtime observation。

2

可选环境刷新

第 0 步如 skill 要求 environment outputs，则通过 observe_for_environment() 和 PerceptionRuntime.refresh_environment() 更新环境状态。

3

PolicyAdapter.to_policy_input()

把 PhyAgentOS runtime observation 映射为 SkillRuntime 所需输入，并注入 task、action_dim、source observation id 等。

4

policy_client.infer()

policy client 经 websocket 发送 msgpack observation，并要求响应包含 actions。

5

PolicyAdapter.from_policy_output()

把 policy 输出统一成 runtime action chunk；缺少动作缓冲会触发 protocol error。

6

target_handle.action_chunk()

依次执行 safety/action bridge、target adapter 的 executable action 转换，然后调用 target 的 chunk 执行接口。

7

终止判断

根据 success、done 或 max_steps 返回结果，并记录 reward、policy latency、final status。

关键边界

Policy runtime 不直接拿 raw target

SkillRuntime 只接收 TargetSessionHandle，所有 observe、action chunk、environment refresh 都经过 handle 统一更新 heartbeat、step index 和 last status。

  • 观察路径：target raw observation → target adapter → runtime observation → policy adapter → policy input。
  • 动作路径：policy output → policy adapter → action bridges → target adapter → target action chunk。

Watchdog 负责外层生命周期

RunnerThreadHandle 在线程中运行 session，并由 watchdog 轮询快照与 execute_timeout_s。超时时请求 cancel/close，正常结束时写入 episode、history、result。

  • policy client 的连接和关闭由 supervisor 管理。
  • runtime compatibility preflight 是进入真实执行前的最后一道契约检查。
